1 Project Overview

What is Aakar?

Aakar (आकार — meaning "shape" in Hindi) is a cross-platform mobile and desktop application designed to help children on the Autism Spectrum learn to recognize, understand, and express emotions. It serves as a comprehensive Autism Assistive Kit for Awareness and Recognition.

🎯
Core Mission: Make emotion education interactive, gamified, and measurable for children with ASD — while giving teachers, therapists, and parents the tools to track progress collaboratively.

Problem Statement

Children with ASD often face three core challenges:

  • Recognizing facial emotions in others
  • Expressing their own emotions appropriately
  • Engaging with traditional, non-interactive learning methods

Aakar addresses these by combining AI-powered emotion detection, real-time live feedback, and therapeutic game-based learning — all in one platform.

4 Workflow & Data Flow

4.1 — User Login & Registration

1
App Launch

Splash screen animates for 2s → checks stored session

2
Role Selection

User picks their role: Child / Teacher / Therapist / Parent

3
Sign Up

Enter name, email, password → password passed through bcrypt hash → stored in SQLite users table

4
Login

Entered password hashed → compared against stored bcrypt hash → session created in AuthProvider

5
Dashboard

Role-based router navigates to the correct dashboard shell

4.2 — Emotion Detection (Cloud Mode)

1
Image Input

User takes photo via camera or picks from gallery

2
Compression

Image resized to 512×512 and JPEG-compressed to ≤500 KB

3
API Call

POST /analyze-emotion — multipart image sent to FastAPI server

4
Face Crop

OpenCV Haar Cascade detects face → crops to 224×224 px

5
DeepFace Analysis

DeepFace.analyze() with MTCNN detector → returns 7 emotion probabilities

6
Post-Processing

Neutral penalty (×0.3) + Fear/Surprise correction + linear normalization applied

7
Save & Display

Result saved to SQLite emotion_logs table and EmotionHistoryProvider → UI shows emoji, confidence bars, "tends between" card

4.3 — Live Detection (WebView Mode)

1
WebView Opens

Flutter loads emotion_detection.html in an embedded WebView

2
Library Injection

Flutter reads face-api.min.js from assets → encodes as base64 → executes in WebView as blob URL script

3
Model Injection

Flutter reads model weight files from assets → encodes as base64 → injects into window.__modelData

4
Fetch Polyfill

Custom window.fetch override intercepts model file requests → serves from __modelData instead of network

5
Detection Loop

Camera starts → every 400ms: TinyFaceDetector finds face → FaceExpressionNet classifies expression → draws bounding box + emotion bars

6
JS → Flutter Bridge

EmotionChannel.postMessage(JSON) sends each result back to Flutter which saves it to EmotionHistoryProvider

4.4 — Teacher–Student Data Flow

StepActionDatabase Operation
1Teacher opens Student Profiles tabgetStudentsByTeacher(teacherId) — JOIN users + mapping table

8 Design Decisions & Best Practices

Architecture Decisions

DecisionWhy
Hybrid detection (device + cloud)On-device TFLite works offline. Cloud DeepFace gives higher accuracy. User toggles via Settings.
SQLite over FirebaseNo internet requirement. Children's data stays on-device (privacy). Zero server configuration.
Provider over BLoC/GetXFlutter team's recommendation. 8 providers are manageable. BLoC would add unnecessary complexity.
face-api.js in WebViewReal-time detection without piping every camera frame from Flutter to Python backend.
Base64 model injectionEliminates CDN/network dependency. Models loaded from Flutter assets → injected into WebView memory.
Role-based separate dashboardsEach role has fundamentally different needs. Separate shells prevent UI complexity and confusion.

Security Practices

PracticeImplementation
Password securitybcrypt hashing with random salt — never stored as plaintext
Local data storageAll user and child data stored in on-device SQLite — not sent to cloud
Camera permissionsRequested at runtime via permission_handler with graceful denial handling
Production APIHTTPS + SSL/TLS, restricted CORS, rate limiting for EC2 deployment

Performance Optimizations

OptimizationDetails
Image compressionResized to 512×512, JPEG-compressed to ≤500 KB before sending to backend
Detection throttlingLive detection runs every 400ms — not per-frame — to prevent CPU overload
Neutral penaltyFER2013 is biased toward "Neutral" — 0.3× multiplier applied to reduce this bias
Fear/Surprise correctionHeuristic correction applied as the model often confuses these two emotions
Const constructorsUsed throughout Flutter widgets to minimize unnecessary rebuilds
